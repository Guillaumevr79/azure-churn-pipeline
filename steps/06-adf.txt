# Orchestration ADF — Pipelines, CI/CD Git et Monitoring

## 1. Prérequis

- ADF adf-churn-pipeline provisionné par Terraform (voir 01-terraform.txt)
- Cluster Databricks churn-cluster en état RUNNING (voir 03-databricks.txt)
- Token d'accès Databricks disponible (Settings → Developer → Access tokens → Generate new token)
- Dépôt GitHub azure-churn-pipeline existant avec branche main

## 2. Intégration Git (GitHub)

Dans l'UI ADF → Manage → Git configuration → Configure :

  Repository type   : GitHub
  GitHub account    : <GITHUB_USERNAME>
  Repository name   : azure-churn-pipeline
  Collaboration branch : main
  Publish branch    : adf_publish     # Branch créée automatiquement par ADF au premier Publish
  Root folder       : /adf            # Les fichiers JSON ADF sont versionnés dans ce dossier

IMPORTANT : choisir "Sélectionner un dépôt" (liste déroulante) et non coller l'URL du dépôt.
Coller l'URL provoque une erreur serveur 500 dans l'UI ADF.

Après configuration, ADF crée automatiquement le dossier /adf dans le dépôt au premier Publish.

## 3. Créer le Linked Service Databricks

Dans ADF → Manage → Linked services → New → Azure Databricks :

  Nom            : AzureDatabricks
  Workspace URL  : https://adb-7405608294355726.6.azuredatabricks.net
  Auth type      : Access token
  Token          : <DATABRICKS_ACCESS_TOKEN>    # Ne pas stocker en clair — utiliser ADF Key Vault LS
  Cluster type   : Existing interactive cluster
  Cluster ID     : <CLUSTER_ID>                 # ex : 0607-174937-h30nuwtu

Tester la connexion avant de sauvegarder.

## 4. Créer les pipelines

### Pipeline pl_ingest_telco

Activité : Notebook Databricks
  Nom        : ingest_telco
  Notebook   : /churn-pipeline/00_ingest_telco
  Linked service : AzureDatabricks

NE PAS laisser de lignes de paramètres vides dans l'onglet "Settings → Base parameters" —
cela provoque une erreur de validation au moment du debug/trigger.

### Pipeline pl_transform_databricks

3 activités Notebook Databricks en séquence :
  1. bronze_to_silver → /churn-pipeline/01_bronze_to_silver
  2. silver_to_gold   → /churn-pipeline/02_silver_to_gold
  3. churn_model      → /churn-pipeline/03_churn_model

Relier les activités par des flèches de dépendance (condition : Success) :
  bronze_to_silver → silver_to_gold → churn_model

IMPORTANT : les activités doivent être explicitement connectées par des flèches dans le canvas.
Sans flèches, ADF les exécute en parallèle (pas en séquence) ce qui provoque des erreurs
car chaque notebook dépend du résultat du précédent.

### Pipeline pl_churn_pipeline (orchestrateur)

2 activités Execute Pipeline en séquence :
  1. run_ingest     → Execute Pipeline : pl_ingest_telco
  2. run_transform  → Execute Pipeline : pl_transform_databricks

Relier par une flèche Success : run_ingest → run_transform

## 5. Publier les pipelines

Dans ADF Studio → bouton Publish (barre supérieure) :

Cette action :
  - Valide tous les pipelines et linked services
  - Déploie sur le service ADF live
  - Pousse les fichiers JSON dans la branche adf_publish du dépôt GitHub (/adf)

Ne pas confondre "Save" (sauvegarde dans la branche de travail) et "Publish" (déploiement live).

## 6. Créer un trigger planifié

Dans ADF → Manage → Triggers → New :

  Nom       : trigger_weekly_churn
  Type      : Schedule
  Récurrence : Toutes les semaines (ex : lundi à 06h00 UTC)
  Pipeline  : pl_churn_pipeline

Activer le trigger après création → Publish pour persister.

## 7. Monitoring

Onglet Monitor dans ADF Studio :

  Pipeline runs   : historique des exécutions, statut, durée
  Activity runs   : détail de chaque activité dans un pipeline
  Trigger runs    : historique des déclenchements

En cas d'erreur, cliquer sur l'activité échouée → icône lunettes → voir les logs Databricks.

Les erreurs fréquentes dans pl_transform_databricks :
  - ModuleNotFoundError: mlflow   → mlflow n'est pas installé comme library sur le cluster
  - SparkConnectGrpcException     → DBR 15.4 utilisé au lieu de 13.3 (voir 03-databricks.txt)
  - Notebook not found            → notebook non importé ou chemin incorrect (sensible à la casse)

## 8. Versionner les artefacts ADF

Après chaque Publish, ADF pousse automatiquement dans /adf sur adf_publish :
  /adf/pipeline/pl_ingest_telco.json
  /adf/pipeline/pl_transform_databricks.json
  /adf/pipeline/pl_churn_pipeline.json
  /adf/linkedService/AzureDatabricks.json
  /adf/trigger/trigger_weekly_churn.json

Pour suivre les changements dans le dépôt :
  git fetch origin adf_publish
  git diff main..adf_publish -- adf/

## Points d'attention

- CRITIQUE — Intégration Git : choisir "Sélectionner un dépôt" (dropdown GitHub OAuth) et non
  coller l'URL manuellement. L'URL manuelle déclenche une erreur 500 côté ADF sans message utile.

- CRITIQUE — Flèches de dépendance obligatoires : sans flèche entre les activités Notebook,
  ADF les parallélise. 02_silver_to_gold lit le Delta écrit par 01_bronze_to_silver —
  une exécution parallèle lève une erreur "path does not exist" ou lit des données vides.

- CRITIQUE — Lignes de paramètres vides : dans l'onglet Settings d'une activité Notebook,
  supprimer toutes les lignes de paramètres non renseignées. ADF les interprète comme des
  paramètres avec une valeur vide et lève une erreur de validation au debug.

- mlflow doit être installé comme library sur le cluster (cluster-scoped, pas notebook-scoped)
  pour que le notebook 03_churn_model fonctionne en mode ADF. Ajouter la library depuis
  l'UI Databricks : Compute → churn-cluster → Libraries → Install new → PyPI → mlflow.
  Un %pip install dans le notebook est insuffisant en contexte ADF job.

- Le token Databricks dans le Linked Service a une durée de vie limitée (max 730 jours).
  Penser à le renouveler avant expiration pour éviter une interruption des pipelines.

- Les fichiers dans adf_publish ne doivent pas être édités manuellement — ils sont
  régénérés à chaque Publish et les modifications manuelles seraient écrasées.
