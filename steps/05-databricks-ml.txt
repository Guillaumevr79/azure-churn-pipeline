## 2. Notebook 03_churn_model.py

Chemin workspace : /churn-pipeline/03_churn_model

Structure du notebook (dans l'ordre des cellules) :

### Cellule 1 — Patch typing_extensions (ne pas modifier)

Le runtime DBR 13.3 livre une version de typing_extensions qui peut manquer Sentinel et NoExtraItems.
Le patch est appliqué avant l'import de mlflow pour éviter les AttributeError au chargement.
Après le patch, les modules mlflow déjà partiellement chargés sont purgés de sys.modules
puis reimportés proprement.

# Ce bloc doit rester en première cellule — avant tout autre import
import sys, typing_extensions
# ... (patch Sentinel + NoExtraItems + purge sys.modules mlflow)

## Points d'attention

- Ne jamais faire %pip install mlflow, numpy ou scikit-learn dans ce notebook.
  Ces packages sont dans le runtime DBR 13.3. Un pip install déclenche un conflit sur
  typing_extensions (attribut Sentinel absent) et numpy (enum _CopyMode absent),
  ce qui rend mlflow non importable même après restartPython().

- restartPython() coupe l'exécution du notebook au point d'appel : tout ce qui suit
  dans la même cellule n'est pas exécuté. Si une cellule contient restartPython(),
  les imports post-restart doivent être dans la cellule suivante.

- Le patch typing_extensions dans la cellule 1 est un contournement du bug de compatibilité
  entre la version de typing_extensions livrée par DBR 13.3 et mlflow. Il doit rester
  en tête de notebook et ne pas être supprimé.

- L'expérience MLflow est créée automatiquement au premier appel de set_experiment().
  Si le chemin /churn-pipeline n'existe pas dans le workspace, le créer d'abord :
    databricks workspace mkdirs /churn-pipeline

- mlflow.sklearn.log_model() enregistre le modèle en artefact du run MLflow, pas dans
  le Model Registry. Pour le déploiement, enregistrer le modèle dans le Registry via :
    mlflow.register_model(f"runs:/<RUN_ID>/churn-model", "ChurnModel")
